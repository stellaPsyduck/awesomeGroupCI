\subsubsection*{Multi-Tasking Extra Trees (or Extremely Randomized Trees)}
Extra Trees (ET) is a tree ensemble method like Random Forests, but it produces more decorrelated trees. It uses
 the entire data set to build decision trees, rather than bootstrapping as in random forests. It also differs
in its threshold selection, which randomly selects where to split the feature rather than using an optimization
method. It still chooses $m_{try}$ number of random features to choose the feature splits to compare, where
$m_{try} = \lfloor \frac{p}{3} \rfloor$ or $m_{try} = \lfloor \sqrt{p} \rfloor$ and p is the number of features. There are two relevant extensions to this algorithm that make it similar
to our LSTM hard parameter multi-tasking \cite{rfmulti}. \\
The first is introducing multi-output, which essentially means the loss function is the summation of sample error
and task outputs. This effectively makes each node split
consider the error of the candiate splits' effect across all tasks \cite{rfmulti}.
The second is introducing task-focused splits, or the ability to make decisions based on what the task is
being predicted. To do this, a random group of $m_{try}$ features and S random possible
subsets of tasks are selected (S$\epsilon \mathbb{Z} < T$ randomly selected). Each feature split is randomized,
and the multi-output error is calculated. Then, $f_k$ (Equation \ref{fk}) is calculated for each candidate task subset or feature;
this is the prediction. The error is calculated from the prediction and used to choose the best split at the node \cite{rfmulti}. 

\begin{equation}
    \label{fk}
    f_k = \frac{\sum_{\nu \epsilon I_k} y_{\nu} + \alpha \frac{1}{\lfloor I \rfloor} \sum_{w \epsilon I} y_w}{\lfloor I_k \rfloor + \alpha}
\end{equation}

\noindent\textit{``I denotes the set of sample indexes at the current node, $I_k$ the set of sample indexes of the task $k$
at the current node and $\alpha$ is a regularization parameter weighting the influence of the tasks with default
value 1. Then, the task cut point is randomly selected from the interval $(\min_k f_k, \max_k f_k)$" - Schmid et. al \cite{rfmulti}}

To implement this method, we used R, since there is a pre-made R package for multi-tasking tree enssemblers. It is technically archived in newer versions
of R, but can be downloaded \href{https://cran.r-project.org/src/contrib/Archive/extraTrees/}{here}. This was decided because the paper explaining the metholody
used this package \cite{rfmulti}.

\subsubsection*{Mean Absolute Error (Criterion and Metric)}
Mean Absolute Error (MAE) is a metric robust to outliers \cite{slater}. 
For that reason, the MAE was chosen as the loss function, and the models
were evaluated with this metric. The MAE could not be used as the “criterion”
or error to minimize in extra trees because the R package did not support it.

\subsection{Preprocessing}
\subsubsection*{Misaligned Dates}
Due to different market holidays, some days in the data included both null values and
recorded prices. The days with null values were removed to keep the dataset consistent.
Dropping a few days of data should not obscure the stocks’ growth patterns and relationships.
\subsubsection*{Logarithmic Scale}
The data was scaled logarithmically. This was to ensure stable predictions by reducing 
 effect of outliers on the model and to account for the exponential patterns in each stock's data.
The predictions become more stable because most analysis techniques assume additive data, including
machine learning.\cite{log_scale}. \\
The scaling affect is shown in appendix \ref{appendix_a}, the medium stock ACIW is on a similar scale to the large 
stock AAPL after logging the data.
\subsubsection*{Lag Length}
Lags are data from a previous date; for example, lag 1 at time $t$ is the data point at time $ t-1$ \cite{slater}.
This idea is from time series, where it is assumed that lags of the data have some affect on the current $t$'s data. \\
For this model, 60 lags were used because a quarter of the fiscal year is approximately this many days \cite{stockgro}. This value considers
only days when the markets are actually open.

\subsubsection*{LSTMs}
Pytorch requires LSTM inputs to be 3D tensors \cite{3d_tensor_lstm}. The data looks at stocks, with the only attribute 
being the stock price. Referring to figure \ref{lstm_diagram}, the matrix on the left would be an example 3D tensor input,
with each vector $\vec{x}$ representing all the stocks of interest at the given t. There is no third dimension depicted,
as the only attribute of each stock of interest is the given price at time t. \\
The times within the input are $\vec{x_{t,n}}$ and the lags of $\vec{x_{tn}}$, which represent our rolling window,
 since $\vec{x_{t+1}}$ should use all the previous lags for its prediction.
\subsubsection*{Extra Trees}
To turn the stock price data into an acceptable input for a tree ensemble, the lags must be added manually.
Each datapoint will then have 60 attributes representing the 60 associated lags
lags \cite{timeseries_data}. Days without all 60 previous lags have been dropped.

\subsubsection*{Extra Trees}
For Extra Trees, single-task and multi-task, the parameters that could be tuned  are \cite{rfmulti}:

\begin{itemize}
\item The number of grown trees
\item The cardinality of the sample of features at every node is $m_{try}$
\item The number of random split values at each node
\item The minimum number of observations that each terminal node should contain (stopping criteria)
\end{itemize}

The models in this paper are tuned for $m_{try}$, the number of random splits/cuts, and the number
of terminal node observations. The number of trees was set to the default of 500 \cite{extratreedoc}.
Any excess grown tree doesn't impact the predictive power of the model, and 500 is likely enough for
the tasks of interest \cite{toomanytree}.

\subsection*{Future Works}
To further this project, the remaining stocks of interest could have their 
single-task counterparts calculated and compared. This would provide a better 
understanding of how multitasking affects the predictions.  \\
Tthe intermediate step of multi-input single-output predictions could also be explored.
This would reveal the effect of shared weights in multi-tasking, because the multi-input
single-output model would use the same input data as our multi-tasking model, but it would optimize for only one prediction.
